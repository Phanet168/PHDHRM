# ១១) ជំនួយលម្អិត៖ ទំព័រគ្រប់គ្រងរចនាសម្ព័ន្ធអង្គភាព

ឯកសារនេះសម្រាប់ការងារ **នៅក្នុងទំព័រ Org Structure នេះប៉ុណ្ណោះ**។

ឯកសារនេះពិពណ៌នាការងារក្នុងផ្ទាំងនេះតែប៉ុណ្ណោះ។

## 11.1 គោលបំណង

ផ្នែកនេះជួយឱ្យអ្នកអាច:

1. កំណត់ទំនាក់ទំនងរវាងប្រភេទអង្គភាព និងតំណែង
2. ចាត់តួនាទីអ្នកប្រើតាមអង្គភាពជាក់លាក់
3. កំណត់សិទ្ធិអំពើការងារតាមតួនាទីអង្គភាព
4. កំណត់លំហូរ Workflow តាម policy
5. រក្សាទុកទិន្នន័យចាស់សម្រាប់ backward compatibility

សេចក្តីបញ្ជាក់៖ គោលបំណងខាងលើអនុវត្តតែសម្រាប់ផ្ទាំងដែលមានក្នុងទំព័រនេះ។

ឧទាហរណ៍ខ្លី៖

- អ្នកចង់ឱ្យប្រធានអង្គភាពអាចអនុម័តលិខិតបាន តែបុគ្គលិកទូទៅមិនអាចអនុម័តបាន
- អ្នកចង់ផ្លាស់ប្តូរបុគ្គលិកពីអង្គភាព A ទៅ B ដោយមិនបាត់ប្រវត្តិចាស់

## 11.2 Quick Start (ចាប់ផ្តើមឆាប់រហ័ស)

បើអ្នកចង់ឱ្យប្រព័ន្ធដំណើរការត្រឹមត្រូវភ្លាម សូមធ្វើតែ 6 ជំហាននេះជាមុន៖

1. បង្កើត/ពិនិត្យ `Responsibilities` ឱ្យបានគ្រប់
2. កំណត់ `Org Position Matrix` តាមប្រភេទអង្គភាព
3. ចាត់តាំងអ្នកប្រើនៅ `User Assignments` (Active + Primary)
4. កំណត់សិទ្ធិ Action នៅ `Org Role Permission Matrix`
5. កំណត់ជំហានអនុម័តនៅ `Workflow Policy Matrix`
6. ពិនិត្យផ្ទៀងផ្ទាត់ជាមួយ `Legacy Org Roles` (សម្រាប់ compatibility)

លទ្ធផលដែលគួរបាន៖ User ម្នាក់ៗឃើញតែសកម្មភាពដែលសមនឹងតួនាទីនៅក្នុងទំព័រ Org Structure និង Workflow មិនរំលងជំហាន។

## 11.3 ម៉ឺនុយក្នុងផ្ទាំងគ្រប់គ្រងរចនាសម្ព័ន្ធអង្គភាព

នៅលើផ្ទាំងនេះ អ្នកនឹងឃើញម៉ឺនុយសំខាន់ៗខាងក្រោម:

1. Org Position Matrix
2. User Assignments
3. Legacy Org Roles
4. Workflow Policy Matrix
5. Org Role Permission Matrix
6. Responsibilities
7. Help

ចំណាំ៖ ទាំងនេះជាផ្នែកដែលធ្វើការជាមួយគ្នា មិនមែនស្ទួនគ្នាទេ។

## 11.4 មុខងារនីមួយៗក្នុងទំព័រនេះ (យល់ងាយ និងអនុវត្តបាន)

### A) Org Position Matrix

ប្រើសម្រាប់កំណត់ថា ប្រភេទអង្គភាពណា អាចមានតំណែងអ្វីខ្លះ។

គួរកំណត់:

1. Unit Type
2. Position/Role ក្នុងអង្គភាព
3. លំដាប់ឋានានុក្រម (បើមាន)
4. សញ្ញាសម្គាល់តំណែងដឹកនាំ (leader/can approve)

ពេលណាត្រូវកែ៖

1. បង្កើតប្រភេទអង្គភាពថ្មី
2. បន្ថែមតំណែងថ្មីតាមរចនាសម្ព័ន្ធ
3. កែគោលនយោបាយអនុម័ត

ឧទាហរណ៍៖

- Unit Type = "នាយកដ្ឋាន" អាចមានតំណែង៖ ប្រធាន, អនុប្រធាន, មន្ត្រី
- Unit Type = "ការិយាល័យ" អាចមានតំណែង៖ ប្រធានការិយាល័យ, អនុប្រធានការិយាល័យ, មន្ត្រី

### B) User Assignments

នេះជាតារាងស្តង់ដារ (canonical) សម្រាប់ចាត់អ្នកប្រើទៅអង្គភាព។

កំណត់សំខាន់៖

1. User
2. Department/Org Unit
3. Responsibility
4. Effective From / Effective To
5. Is Active
6. Is Primary

ច្បាប់អនុវត្ត៖

1. User ម្នាក់គួរមាន Primary សកម្មតែមួយ
2. កាលបរិច្ឆេទមានប្រសិទ្ធភាពត្រូវមិនជាន់គ្នាខុសតួនាទី
3. មុន disable assignment ត្រូវពិនិត្យការងារកំពុងរង់ចាំអនុម័ត

ឧទាហរណ៍៖

- លោក សុភា ជា Primary នៅអង្គភាព IT ចាប់ពី 01-01-2026
- ពេលផ្ទេរទៅអង្គភាព Admin នៅ 01-05-2026 ត្រូវបិទ assignment ចាស់ដោយកំណត់ Effective To = 30-04-2026
- បង្កើត assignment ថ្មីនៅ Admin និងកំណត់ Is Primary = Yes

### C) Legacy Org Roles

ផ្នែកនេះសម្រាប់ទិន្នន័យចាស់ ដើម្បីឱ្យមុខងារចាស់នៅដំណើរការបាន (backward compatibility)។

គោលការណ៍៖

1. មិនលុបទិន្នន័យ legacy ចោលដោយគ្មានផែនការ
2. ប្រើសម្រាប់ត្រួតពិនិត្យ/ផ្ទៀងផ្ទាត់ជាមួយ User Assignments
3. ការសរសេរទិន្នន័យថ្មីគួរផ្អែកលើ User Assignments ជាចម្បង

ឧទាហរណ៍៖

- ផ្ទាំងចាស់មួយនៅតែអានពី Legacy Org Roles ដូច្នេះត្រូវរក្សាទុកទិន្នន័យចាស់ឱ្យនៅ
- ប៉ុន្តែពេលចាត់តាំងថ្មី គួរធ្វើនៅ User Assignments ជាមុន

### D) Workflow Policy Matrix

ប្រើសម្រាប់កំណត់ជំហានអនុម័តតាមប្រភេទសំណើដែលគ្រប់គ្រងពីទំព័រនេះ។

ត្រូវកំណត់៖

1. Module/Request Type
2. លក្ខខណ្ឌ (threshold/condition)
3. ជំហាន (step order)
4. តួនាទីត្រូវអនុម័តក្នុងជំហាននីមួយ
5. Final step

ឧទាហរណ៍៖

- Policy A៖ Manager -> Head
- Policy B៖ Manager -> Deputy Head -> Head

### E) Org Role Permission Matrix

ប្រើសម្រាប់កំណត់ថា តួនាទីអង្គភាពណា អាចធ្វើ action អ្វីលើ module ណា។

ឧទាហរណ៍ action៖

1. create
2. review
3. approve
4. reject
5. delegate

ឧទាហរណ៍ការកំណត់៖

- Head៖ approve, reject, delegate
- Manager៖ review, forward
- Staff៖ create, edit_own

### F) Responsibilities

ជាបញ្ជីតួនាទីស្តង់ដារ (ឧ. head, deputy_head, manager, reviewer, staff) សម្រាប់យកទៅប្រើក្នុង User Assignments និង Policy ផ្សេងៗ។

## 11.5 លំដាប់ការរៀបចំដែលណែនាំ (Setup Order)

សម្រាប់អង្គភាពថ្មី ឬកែរចនាសម្ព័ន្ធថ្មី សូមអនុវត្តតាមលំដាប់នេះ:

1. កំណត់ Responsibilities ឱ្យគ្រប់
2. កំណត់ Org Position Matrix
3. ចាត់ User Assignments (កំណត់ Primary/Active/Effective Date)
4. កំណត់ Org Role Permission Matrix
5. កំណត់ Workflow Policy Matrix
6. ផ្ទៀងផ្ទាត់ជាមួយ Legacy Org Roles (បើមាន)

## 11.6 SOP ប្រចាំថ្ងៃសម្រាប់អ្នកគ្រប់គ្រង (ក្នុងទំព័រនេះ)

### ករណី ១: ចាត់តាំងប្រធានអង្គភាពថ្មី

1. ចូល User Assignments
2. ជ្រើស User និង Org Unit
3. ជ្រើស Responsibility = head
4. កំណត់ Is Active = Yes
5. កំណត់ Is Primary = Yes
6. កំណត់ Effective From ជាថ្ងៃចាប់ផ្តើមអនុវត្ត
7. Save និងត្រួតពិនិត្យថា primary ចាស់ត្រូវបានបិទ/ផុតសុពលភាពរួច

### ករណី ២: ផ្ទេរបុគ្គលិកពីអង្គភាព A ទៅ B

1. កែ assignment ចាស់៖ កំណត់ Effective To ឬបិទ Is Active
2. បង្កើត assignment ថ្មីសម្រាប់អង្គភាព B
3. កំណត់ Primary តាមស្ថានភាពពិត
4. ផ្ទៀងផ្ទាត់ Permission Matrix ថាតួនាទីថ្មីមានសិទ្ធិត្រឹមត្រូវ

ចំណាំ៖ ការណែនាំនេះគ្រាន់តែបង្ហាញការកែ Assignment ក្នុងទំព័រ Org Structure ប៉ុណ្ណោះ។

### ករណី ៣: មិនឃើញប៊ូតុងអនុម័ត

ពិនិត្យតាមលំដាប់៖

1. User មាន assignment សកម្មឬនៅ
2. Responsibility ត្រឹមត្រូវឬនៅ
3. Role នោះមាន action ក្នុង Org Role Permission Matrix ឬនៅ
4. Workflow Policy មាន step ដែលផ្គូផ្គងលក្ខខណ្ឌឬនៅ

ឧទាហរណ៍ពិនិត្យលឿន៖

- បុគ្គលិក B មើលឃើញតែប៊ូតុង "Create" តែគ្មាន "Approve" គឺធម្មតា ប្រសិនបើតួនាទីគាត់ជា Staff
- ប្រសិនបើអ្នកជា Head តែមិនឃើញ "Approve" សូមពិនិត្យ Is Active និង Effective Date ជាមុន

## 11.7 Do និង Don't (អនុវត្តត្រឹមត្រូវក្នុងទំព័រនេះ)

Do៖

1. កំណត់ Primary តែមួយសម្រាប់ User ម្នាក់ក្នុងពេលតែមួយ
2. កំណត់ Effective From/To ឱ្យច្បាស់ពេលផ្ទេរតួនាទី
3. ពិនិត្យ Permission Matrix និង Workflow Policy មុនដាក់ប្រើ

Don't៖

1. កុំកែ Legacy Org Roles ជា source សំខាន់សម្រាប់ទិន្នន័យថ្មី
2. កុំទុក assignment ចាស់ active ជាមួយ assignment ថ្មីដោយគ្មានគោលបំណង
3. កុំបើកសិទ្ធិ approve ជាសកលដោយគ្មានការគ្រប់គ្រង scope

4. កុំច្រឡំយកជំនួយពីទំព័រផ្សេងមកអនុវត្តជំនួសផ្ទាំងនៅទីនេះ

## 11.8 កំហុសញឹកញាប់ និងវិធីដោះស្រាយ (ក្នុងទំព័រនេះ)

1. មានអ្នកអនុម័តម្នាក់ ប៉ុន្តែមិនទទួលការងារ
មូលហេតុ៖ Effective date មិនទាន់ចូលសុពលភាព ឬ Is Active = No

2. អ្នកប្រើម្នាក់មាន primary ច្រើន
មូលហេតុ៖ កំណត់ខុសពេលផ្ទេរទីតាំង

3. អាចចូលម៉ឺនុយបាន ប៉ុន្តែធ្វើ action មិនបាន
មូលហេតុ៖ មាន access កម្រិត menu ប៉ុន្តែខ្វះ org-role action permission

4. Workflow រំលងជំហាន
មូលហេតុ៖ Policy priority/condition កំណត់មិនត្រឹមត្រូវ

5. ព័ត៌មានក្នុង Help មិនដូចអ្វីដែលអ្នករំពឹងពីទំព័រផ្សេង
មូលហេតុ៖ ឯកសារនេះផ្តោតតែផ្ទាំង Org Structure ប៉ុណ្ណោះ

## 11.9 Checklist មុនដាក់ប្រើប្រាស់

1. Responsibilities បានកំណត់ឈ្មោះស្តង់ដារ និងសកម្ម
2. Org Position Matrix ស្របតាមរចនាសម្ព័ន្ធជាក់ស្តែង
3. User Assignments មាន data គ្រប់ និងមាន primary សកម្មម្នាក់មួយ
4. Org Role Permission Matrix គ្រប់ action សម្រាប់ module ចាំបាច់
5. Workflow Policy Matrix មានលំដាប់ជំហានច្បាស់
6. Legacy Org Roles នៅតែមានសុវត្ថិភាពសម្រាប់ backward compatibility

7. បានពិនិត្យថាខ្លឹមសារដែលប្រើសម្រាប់អនុវត្ត គឺស្ថិតនៅក្នុងផ្ទាំងទំព័រនេះពិតប្រាកដ

## 11.10 សេចក្តីសន្និដ្ឋាន

ទំព័រគ្រប់គ្រងរចនាសម្ព័ន្ធអង្គភាព គឺជាគោលស្នូលសម្រាប់កំណត់ថា អ្នកណាធ្វើអ្វី នៅអង្គភាពណា និងតាមលំដាប់ណា។

បើរៀបចំតាមលំដាប់ដែលបានណែនាំខាងលើ ប្រព័ន្ធនឹង:

1. ងាយថែទាំ
2. កាត់បន្ថយកំហុសសិទ្ធិ
3. ងាយស្រួលត្រួតពិនិត្យការអនុម័ត
4. រក្សាបាន backward compatibility ជាមួយទិន្នន័យចាស់

ឧទាហរណ៍លទ្ធផលល្អបន្ទាប់ពីរៀបចំត្រឹមត្រូវ៖

- អ្នកប្រើម្នាក់ចូលប្រព័ន្ធហើយឃើញតែប៊ូតុងដែលសមនឹងតួនាទីរបស់គាត់
- សំណើច្បាប់ឆ្លងជំហានអនុម័តត្រឹមត្រូវ មិនរំលង និងមិនជាប់គាំង

សេចក្តីបញ្ចប់៖ ឯកសារនេះកំណត់តែការងារក្នុងទំព័រនេះប៉ុណ្ណោះ។
